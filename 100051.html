Design

General

The Wistom Application Programmers Interface (API) has been developed to give a generic and versatile interface to all Wistom functions. The API provides a rather small command set that still lets the user read and write almost any value in Wistom. Furthermore, user identities and passwords are used to provide the necessary security features.

The API exists in two forms; either as a text-based command set ( the MMI) or as a binary protocol that can be transferred on any interface (e.g. TCP/IP). The binary representation is described in section 3.2.6.

Begin and End a Session

The first command in an API session must always be a login. The command shall be followed with a user identity and a password. User ids are stored in Wistom with the current password and also the user’s privileges. So a login is compared to the internally stored settings, and the API session is configured correspondingly.

The user privileges are defined with a user level (a number from 1 to 5), and each API operation can only be executed or read by a user with a user level of a certain value (or less).

A session is ended with the logout command (with no parameters). There is no response to the logout command.

Fundamental Operations

The core of the API is just two commands. A get is used to fetch any parameter or data from Wistom, and a set is used to set a parameter, trigger a function or perform an operation. The commands are used to access defined operations in Wistom, which are identified with an application id (app) and an operation id (op). Each app/op pair gives the user access to one or more parameters, functions or data.

The normal responses to get and set are “get result” (getRes) and “set acknowledge” (setAck) respectively. If something went wrong with the get or set, “get error” (getErr) or “set negative acknowledge” (setNAck) is returned instead.

Since multiple get and set commands can be issued simultaneously or in a tight sequence, each command is accompanied with a token. The token is an integer value, which is always returned in the response from Wistom. It is up to the user to employ the tokens wisely, so that it can discern one operation from the other.

Data Format

The data part of an API message is structured in a special way. Since API messages can be both text-based and binary, it is essential to be able to discern how the data is structured, i.e. if the data is composed of integers, floats or just plain text strings. Furthermore, the amount of data for a specific operation may vary with the application, and over time. The selected solution to this has been to “tag” each data element. For each application/operation pair a set of tags is defined. Each tag defines exactly how much data that follows and how it shall be interpreted (e.g. five unsigned integers). The amount of tags and data varies depending on what the user wants to achieve.

Examples

The following examples may further clarify the above discussion on applications, operations, tags and data:

Example 1: Get board temperature

get SMGR TEMP
getRes SMGR TEMP 1 27.3 2 32.5 3 45.1

The command is a get from the application System Manager (“SMGR”), with the operation Temperature (“TEMP”). The response to this is a getRes with the tags “1” and “2”. Both tags are followed by a 32-bit floating-point value that denotes the board temperature and the sensor temperature respectively.

Example 2: Set channel configuration

set SPEC CHNL 2 1 3 1 4 194.1e12 12 25 8 0.001e12
setAck

The command is a set from the application Spectrum Parameters (“SPEC”), with the operation Channel Configuration (“CHNL”) with the tags “2”, “3”, “4”, “12” and “8”. This command will set the channel configuration for channel 1 (tag “2”) to “configured” (tag “3”) with the center frequency 194.1 THz (tag “4”), OSNR warning level to 25 dB (tag “12”) and with the frequency hysteresis 1 GHz (tag “8”). See section 5.5.1 for more information about the channel configuration. Note that channel parameters that are not included in this command are unchanged.

Binary Representation

The Wistom API is by definition in big-endian1 format, i.e. the most significant byte shall be sent first for values that consist of more than one byte. This means that for example on a desktop PC, all data must be switched to little-endian format before it can be readable by a program.

The data field tags are 8-bit integers and the type of the actual data varies but is always defined by the tag value.

Alarms

General

There are two ways to obtain alarms generated by the system, either by polling the system or by subscribing to alarms.

Polling the system is accomplished by using the API command get ALMH ALRM. This request returns current alarm status.

If subscribing to an alarm, an API message is sent to the subscriber when a status change occurs. It is important to understand how this feature is handled by the system in order to set up and maintain a subscription.

To set up a subscription use ALMH SUBS and to terminate subscriptions use ALMH UNSU.

The most important aspect to remember is that a subscription is associated with a particular API session. Therefore, a subscription only lives as long as the API session lives. Also, it is only possible to receive an alarm message and maintain current subscriptions in the system from where the actual subscription was made (API session).

An alarm token of choice is given to each subscription (this is not the same as the API message token). The alarm token has a dual purpose. First of all, it is returned in all subsequent API alarm messages. Secondly, it is used as an identifier when terminating a subscription or bundled subscriptions. By giving several subscriptions the same token, the subscriptions will be bundled together and can be unsubscribed with one operation.

NOTE! Since a subscription is associated with an API session it is possible for two different subscriptions from different API sessions to have the same token without any risk of interference.

There are four different API alarm messages. Depending on what time format is used (ALMH TITY), the corresponding message is returned.

NOTE! If the time format is changed all subsequent API alarm messages will also change.
