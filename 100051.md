#  API

## Old and new hardware compatibiity

This manual is valid for the new, Linux based, hardware, WEB2017, it is
also mostly valid for the old hardware platform WEB2007 as the purpose
has been to retain backward compatibility. However, there are
differences and it is recommended to use revision K of this document
when information is needed for WEB2002. There are also difference as
WEB2017 has a 8 times higher sampling rate, higher ADC resolution and
support more ports.

## Overview

This document has been updated with the modification included for the
WistSense app and the updates from the switch to a new hardware platform
WEB-2017 using a Linux instead of OSE based platform

The API consists of the commands and responses outlined in Table 1. More
detailed descriptions can be found in section 3.2.

  ------------- ---------------- --------------------------------------------
  **Command**   **Response**     **Description**

  Login         LoginResult      Starts a session. The command holds user id
                                 and password. The response holds the user
                                 access rights or an error code.

  Logout        \-               This ends a session. There is no response
                                 since all instances immediately terminates.

  Get           GetRes/GetErr    Get a setting, a value, or data from WISTOM.

  Set           SetAck/setNAck   Set a setting or a value in WISTOM. This
                                 could also be a subscription for alarms or
                                 data, or it could trigger an operation.

  (get)         Alarm            Alarm message, is sent spontaneously (after
                                 subscription) or triggered by a get request
  ------------- ---------------- --------------------------------------------

## Design

### General

The W[istom]{.smallcaps}^^ Application Programmers Interface (API) has
been developed to give a generic and versatile interface to all
W[istom]{.smallcaps}^^ functions. The API provides a rather small
command set that still lets the user read and write almost any value in
W[istom]{.smallcaps}^^. Furthermore, user identities and passwords are
used to provide the necessary security features.

The API exists in two forms; either as a text-based command set ( the
MMI) or as a binary protocol that can be transferred on any interface
(e.g. TCP/IP). The binary representation is described in section 3.2.6.

### Begin and End a Session

The first command in an API session must always be a *login*. The
command shall be followed with a user identity and a password. User ids
are stored in W[istom]{.smallcaps}^^ with the current password and also
the user's privileges. So a login is compared to the internally stored
settings, and the API session is configured correspondingly.

The user privileges are defined with a user level (a number from 1 to
5), and each API operation can only be executed or read by a user with a
user level of a certain value (or less).

A session is ended with the *logout* command (with no parameters). There
is no response to the logout command.

### Fundamental Operations

The core of the API is just two commands. A *get* is used to fetch any
parameter or data from W[istom]{.smallcaps}^^, and a *set* is used to
set a parameter, trigger a function or perform an operation. The
commands are used to access defined *operations* in
W[istom]{.smallcaps}^^, which are identified with an *application id*
(app) and an *operation id* (op). Each app/op pair gives the user access
to one or more parameters, functions or data.

The normal responses to *get* and *set* are "get result" (*getRes*) and
"set acknowledge" (*setAck*) respectively. If something went wrong with
the get or set, "get error" (*getErr*) or "set negative acknowledge"
(*setNAck*) is returned instead.

Since multiple *get* and *set* commands can be issued simultaneously or
in a tight sequence, each command is accompanied with a *token*. The
*token* is an integer value, which is always returned in the response
from W[istom]{.smallcaps}^^. It is up to the user to employ the tokens
wisely, so that it can discern one operation from the other.

### Data Format

The data part of an API message is structured in a special way. Since
API messages can be both text-based and binary, it is essential to be
able to discern how the data is structured, i.e. if the data is composed
of integers, floats or just plain text strings. Furthermore, the amount
of data for a specific operation may vary with the application, and over
time. The selected solution to this has been to "tag" each data element.
For each application/operation pair a set of *tags* is defined. Each tag
defines exactly how much data that follows and how it shall be
interpreted (e.g. five unsigned integers). The amount of tags and data
varies depending on what the user wants to achieve.

The data types that are used in the W[istom]{.smallcaps}^^ API are
listed in Table 2.

+----------------------------+-----------------------------------------+
| -   Type                   | -   Description                         |
+============================+=========================================+
| **INT8, INT16, INT32**     | Signed integer. 8, 16 and 32 denotes    |
|                            | the number of bits in the integer. E.g. |
|                            | INT32 for a 32-bit signed integer.      |
+----------------------------+-----------------------------------------+
| **UINT8, UINT16, UINT32**  | Unsigned integer. 8, 16 and 32 denotes  |
|                            | the number of bits in the integer. E.g. |

+----------------------------+-----------------------------------------+

+----------------------------+-----------------------------------------+

### Examples

The following examples may further clarify the above discussion on
applications, operations, tags and data:

**Example 1: Get board temperature**

get SMGR TEMP\
getRes SMGR TEMP 1 27.3 2 32.5 3 45.1

The command is a *get* from the application System Manager ("SMGR"),
with the operation Temperature ("TEMP"). The response to this is a
*getRes* with the tags "1" and "2". Both tags are followed by a 32-bit
floating-point value that denotes the board temperature and the sensor
temperature respectively.

**Example 2: Set channel configuration**

set SPEC CHNL 2 1 3 1 4 194.1e12 12 25 8 0.001e12\
setAck

The command is a *set* from the application Spectrum Parameters
("SPEC"), with the operation Channel Configuration ("CHNL") with the
tags "2", "3", "4", "12" and "8". This command will set the channel
configuration for channel 1 (tag "2") to "configured" (tag "3") with the
center frequency 194.1 THz (tag "4"), OSNR warning level to 25 dB (tag
"12") and with the frequency hysteresis 1 GHz (tag "8"). See section
5.5.1 for more information about the channel configuration. Note that
channel parameters that are not included in this command are unchanged.

### Binary Representation

The W[istom]{.smallcaps}^^ API is by definition in *big-endian*[^1]
format, i.e. the most significant byte shall be sent first for values
that consist of more than one byte. This means that for example on a
desktop PC, all data must be switched to little-endian format before it
can be readable by a program.

The data field tags are 8-bit integers and the type of the actual data
varies but is always defined by the tag value.

## Alarms

### General

There are two ways to obtain alarms generated by the system, either by
polling the system or by subscribing to alarms.

Polling the system is accomplished by using the API command get ALMH
ALRM. This request returns current alarm status.

If subscribing to an alarm, an API message is sent to the subscriber
when a status change occurs. It is important to understand how this
feature is handled by the system in order to set up and maintain a
subscription.

To set up a subscription use ALMH SUBS and to terminate subscriptions
use ALMH UNSU.

The most important aspect to remember is that a subscription is
associated with a particular API session. Therefore, **a subscription
only lives as long as the API session lives**. Also, it is only possible
to receive an alarm message and maintain current subscriptions in the
system from where the actual subscription was made (API session).

An alarm token of choice is given to each subscription (this is not the
same as the API message token). The alarm token has a dual purpose.
First of all, it is returned in all subsequent API alarm messages.
Secondly, it is used as an identifier when terminating a subscription or
bundled subscriptions. By giving several subscriptions the same token,
the subscriptions will be bundled together and can be unsubscribed with
one operation.

**NOTE!** Since a subscription is associated with an API session it is
possible for two different subscriptions from different API sessions to
have the same token without any risk of interference.

There are four different API alarm messages. Depending on what time
format is used (ALMH TITY), the corresponding message is returned.

**NOTE!** If the time format is changed all subsequent API alarm
messages will also change.

## Detailed command descriptions

The table columns have the following meanings:

**Application** The application that this table references. Same as the
section header. The application code is the code or value that is used
to reference this application.

**Operation** The name of the operation that is described in this table
row(s). The *operation code* is the code or value that is used to
reference this application.

**Max. User Lvl** The maximum allowed user level (section 4) that is
required to "get" or "set" this operation. A "-" (minus sign) means that
the operation is not allowed for this command.

**Data Field** The data field consists of "tags" with corresponding
values. The tags defines what kind of data follows, and its type.

**Data Tag** The tags are 8-bit values that are unique within this
operation.

**Data type** The type of the data item. E.g. 32-bit integer, four
characters, a zero-terminated ASCII string etc.

**Class** Whether this data item is mandatory (M) or optional (O). This
is specified independently for "get", "set" and "getRes" commands
